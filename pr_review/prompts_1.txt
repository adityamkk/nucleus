Output:
1. Verification / Signature Check
def verify(sig, body: bytes) -> bool:
    expected = hashlib.sha256(
        (WEBHOOK_SECRET + body.decode("utf-8")).encode("utf-8")
    ).hexdigest()
    return expected == sig
Issues:

Timing attack vulnerability: Using == for signature comparison can be exploited via timing attacks.

Assumes UTF-8 encoding: body.decode("utf-8") will fail if the webhook sends non-UTF-8 bytes.

No exception handling: Malformed bytes could crash the server.

Suggested fixes:

Use hmac.compare_digest(expected, sig) for constant-time comparison.

Wrap decoding in a try/except to handle malformed input gracefully.

2. JSON / Payload parsing
payload = json.loads(raw.decode("utf-8"))
email = payload.get("email", "")
role = payload.get("role", "user")
Issues:

No type or format checks: email or role could be None, numbers, or even objects instead of strings.

No validation of allowed roles: Any client can send "role": "admin" and it will be accepted.

Missing metadata validation: The code comments indicate metadata is important for checking source == "vendor", but no check is done.

Suggested fixes:

Validate that email is a string and matches a proper email regex.

Validate role against a whitelist (e.g., ["user", "admin"]).

Check if metadata exists, is a dictionary, and if metadata.get("source") == "vendor" before applying any special logic.

3. SQL Injection Risk

cur.execute(
    f"INSERT INTO users(email, role) VALUES('{email}', '{role}')"

Direct string interpolation into SQL: This is classic SQL injection. Any field (email, role, or raw) can contain SQL code.

Suggested fixes:

Use parameterized queries instead of string formatting:

    "INSERT INTO webhook_audit(email, raw_json) VALUES (?, ?)", 
    (email, raw.decode("utf-8"))
)
cur.execute(
    "INSERT INTO users(email, role) VALUES(?, ?)", 
    (email, role)
)
4. Upsert Logic / Duplicate Handling
cur.execute(
    f"INSERT INTO users(email, role) VALUES('{email}', '{role}')"
)
Issues:

If the user already exists, this will fail. There’s no proper upsert logic.

Could allow multiple entries for the same email if not careful.

Suggested fixes:

Use INSERT OR REPLACE or INSERT ... ON CONFLICT(email) DO UPDATE SET role=excluded.role.

Issues:

No length check: Very large payloads could crash the DB.

No sanitization: Special characters in raw JSON could break SQL without parameterized queries.

Suggested fixes:

6. Environment Variable Defaults

WEBHOOK_SECRET = os.getenv("WEBHOOK_SECRET", "dev-secret")  # default for dev
Issues:

Weak default secret: Using "dev-secret" in production is unsafe.

No path validation: DB_PATH could point to sensitive system files if misconfigured.

Suggested fixes:

Require WEBHOOK_SECRET to be set in production, fail startup if not.

Validate DB_PATH to ensure it points to a safe location.

7. General Error Handling

No try/except around database operations. A malformed request could crash the endpoint.

No logging of failures besides returning HTTP 401.

No handling of malformed JSON.

Suggested fixes:

Wrap DB operations in try/except.

Return 400 for malformed JSON, 500 for DB errors.

Log exceptions securely without exposing secrets.

8. Missing HTTP Security Headers / Rate Limiting

The endpoint is open to the internet and only relies on signature.

No rate limiting, CSRF, or other protections.

Suggested fixes:
